SOLID Object Oriented Design

SOLID Object Oriented Design

These principles establish practices that lend to developing software with considerations for maintaining and extending as the project grows. Adopting these practices can also contribute to avoiding code smells, refactoring code, and Agile or Adaptive software development.

SOLID stands for:

    S - Single-responsiblity Principle

    O - Open-closed Principle

    L - Liskov Substitution Principle

    I - Interface Segregation Principle

    D - Dependency Inversion Principle

Single-Responsibility Principle

Single-responsibility Principle (SRP) states:

class should have one and only one reason to change, meaning that a class should have only one job.

In the Calculator example, We have to apply different operators like addition, subtraction, multiply and division.

First, create the Addition classes and have the constructors set up the required parameters.

Next, create the Calculations class and then write up the logic to sum up the numbers.

Open-Closed Principle

Open-closed Principle (OCP) states:

Objects or entities should be open for extension but closed for modification.

This means that a class should be extendable without modifying the class itself.

Calculations class we can extend and add other object like multiply and division or history to extend it. We can without modifying exiting parent classes we can extend calculator and introduce new operator.

Liskov Substitution Principle

This means that every subclass or derived class should be substitutable for their base or parent class.

We have coded every class separately and used the hierarchy method WHich helped us to extend the calculator without causing any negative impact on existing code

Interface Segregation Principle

Make fine grained interfaces that are client specific Clients should not be forced to depend upon interfaces that they do not use. This principle deals with the disadvantages of implementing big interfaces.

In Calculator single class we have implemented several interfaces. So we can provide a single implementation for all the common methods between the interfaces. The segregated interfaces will also force us to think of our code more from the client’s point of view, which will in turn lead to loose coupling and easy testing. So, not only have we made our code better to our clients, we also made it easier for ourselves to understand, test and implement.

Dependency Inversion Principle

Entities must depend on abstractions, not on concretions. It states that the high-level module must not depend on the low-level module, but they should depend on abstractions.

Calculation class is dependent on low level class. If we are looking to add another function to out calculator then we have to introduce another object or update existing object.
